# RsyncWin 專案說明

## 一句話定位

**RsyncWin 是一個用 C# / .NET 10 撰寫的原生 Windows rsync 用戶端**，直接實作 rsync 的線路協定（wire
protocol），讓 Windows 機器能與**標準的 rsync 伺服器**同步檔案，**不需要 WSL、Cygwin 或任何 rsync
二進位檔**。

## 解決什麼問題

rsync 是 Unix/Linux 世界事實上的檔案同步標準，但它並沒有原生的 Windows 版本。長期以來 Windows 使用者
只能靠以下幾種方式，各有痛點：

| 既有作法 | 痛點 |
|---|---|
| WSL 裡跑 Linux rsync | 需要安裝 WSL，跨 WSL/Windows 檔案系統邊界效能差、路徑轉換易出錯 |
| Cygwin / MSYS2 rsync | 需要一整套 POSIX 模擬層，部署笨重，NTFS 語意常對不上 |
| cwRsync 等封裝 | 本質仍是 Cygwin 版 rsync，同上 |

RsyncWin 的作法完全不同：**不移植 rsync 的程式碼，而是重新實作它的協定**。它是一個純 .NET 程式，直接
講 rsync 伺服器聽得懂的位元組，因此：

- 不需要 POSIX 模擬層，就是一個 `rsyncwin.exe`。
- 原生處理 NTFS 現實（`\\?\` 長路徑、保留裝置名、備用資料流冒號、大小寫等）。
- 依賴極少（只有 `System.IO.Hashing` 與 `System.CommandLine`）。

## 範圍（Scope）

**已實作、且與真實 rsync 3.4.3 逐位元組互通：**

- **拉取（pull）**：從遠端 rsync 伺服器把樹狀目錄同步到本機。
- **推送（push）**：把本機樹狀目錄同步到遠端 rsync 伺服器。
- **兩種傳輸方式**：
  - **SSH**：透過內建的 `C:\Windows\System32\OpenSSH\ssh.exe` 連線（`user@host:path`）。
  - **rsync 常駐程式（daemon）**：透過 TCP 873 的 `rsync://` 協定，含匿名與密碼驗證。
- **差異傳輸（delta transfer）**：只傳輸有變動的區塊，其餘從本機基準檔（basis）重建。
- **旗標**：`-r`（遞迴）、`-t`（保留時間）、`-a`（封存，pull）、`-c`/`--checksum`（以內容比對）、
  `--delete`（刪除多餘檔案）、`-s`/`--secluded-args`（保護含空白的遠端路徑）、
  `-z`/`--compress`（zlibx 壓縮）。
- **完整的 rsync 數值離開碼（exit code）**。

**設計上刻意排除（附理由）：**

- 純 CLI，**沒有 GUI**。
- 協定 27/29 只支援到序幕/argv 層級，不解碼 29 世代的檔案清單格式（真 29-only 對端會在檔案清單中途
  失步）。
- 壓縮只支援 **zlibx**：標準 `-z` 預設協商 `zstd`，`--old-compress`（舊 zlib）需要一個 BCL 沒有的
  「把符合區塊塞進壓縮視窗但不輸出」原語（`Z_INSERT_ONLY`）。詳見文件四。
- 推送不支援 `-a` 的額外欄位（uid/gid/符號連結/裝置檔）。
- 大於 2 GiB 的檔案（來源整檔讀入記憶體的限制，尚未串流化）。

## 核心設計決策：純協定核心（pure protocol core）

整個專案最關鍵、也最「承重」的一個決定是：**把所有 I/O 推到邊緣，中間留一個零 I/O 的純協定核心**。

`RsyncWin.Protocol` 這個組件裡**沒有任何 `Process`、`Socket` 或檔案系統呼叫**——它只把位元組轉成資料
結構、把資料結構轉回位元組。這帶來一個決定性的好處：

> **那些「錯一個位元組就靜默全量重傳或 exit 12 失步」的逐位元組公式，可以用離線的黃金向量（golden
> vector）測試來驗證，完全不需要網路或 rsync 二進位檔。**

這就是為什麼本專案能在沒有 Linux 環境的 Windows 上，穩定地開發出一個逐位元組正確的 rsync 用戶端。

## 專案分層

| 專案 | 角色 |
|---|---|
| `src/RsyncWin.Protocol` | **純核心**：`Wire`、`Mux`、`Session`、`Checksums`、`FileList`、`Delta`、壓縮。無 I/O。 |
| `src/RsyncWin.Transport` | `IRsyncTransport` 接縫 → `OpenSshProcessTransport`、`DaemonTcpTransport` |
| `src/RsyncWin.Fs` | 所有 Windows/NTFS 現實：`\\?\` 路徑、檔名淨化、屬性政策 |
| `src/RsyncWin.Engine` | 編排 generator/receiver（pull）與 sender（push），走 Channels |
| `src/RsyncWin.Cli` | 薄殼：解析參數、原封不動回傳 rsync 的數值離開碼 |

架構細節見[文件三：結構與算法說明](03-結構與算法說明.md)。

## 開發方法：為什麼它能互通

rsync 協定的失敗模式是**靜默的**——寫錯一個值不會噴錯，而是變成「整檔重傳」或「exit 12 失步」，甚至
在階段邊界變成**卡住不動**。因此本專案的每個階段都強制走同一套工作流程：

1. **先寫規格**：從行為來源（rsync 是 GPLv3，只讀行為不抄程式碼）推導位元組佈局，寫進 `docs/*-spec.md`。
2. **擷取黃金向量**：在**用完即丟的 Docker 容器**裡跑真的 rsync 3.4.3，擷取雙向線路位元組。**絕不在
   Windows 主機上安裝 rsync。**
3. **用擷取的位元組逐位元組驗證規格**。
4. **先實作純核心**。
5. **用擷取向量重播做隱密測試（hermetic test）**——這是主要的正確性閘門，不需要網路。
6. **用真 rsync 做一次端對端 live 互通測試**（Docker 容器），每個測試都有卡住偵測逾時。
7. **對抗式審查**（協定/Windows 檔案系統/測試強度三個視角）。
8. **在階段斷點提交，並在同一個提交更新文件。**

驗收閘門永遠是「位元組/雜湊相等」或「重跑不傳輸任何東西」，**絕不是「看起來對」**。

## 授權與來源規範

標準 rsync（[RsyncProject/rsync](https://github.com/RsyncProject/rsync)）是 **GPLv3**，本專案只讀它的
**行為**。純量協定常數是「事實」，可以記錄；任何比純量大的東西（varint 表、checksum 表）都是「表達」，
必須從寬鬆授權的參考實作（[openrsync](https://github.com/kristapsdz/openrsync) ISC/BSD、
[gokrazy/rsync](https://github.com/gokrazy/rsync) BSD-3）移植，或依文件化演算法重新生成，**絕不從
`rsync.h` 複製**。
