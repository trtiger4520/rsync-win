# RsyncWin 結構與算法說明

本文說明 RsyncWin 的程式結構與核心演算法。目標讀者是想理解「這個 rsync 用戶端內部怎麼運作」的工程師。

---

## 一、整體架構：純核心 + 邊緣 I/O

RsyncWin 分成五個組件，最重要的原則是 **`RsyncWin.Protocol` 純核心不做任何 I/O**：

```
┌─────────────────────────────────────────────────────────────┐
│  RsyncWin.Cli        薄殼：解析參數、對應離開碼               │
├─────────────────────────────────────────────────────────────┤
│  RsyncWin.Engine     編排：PullSession / PushSession /        │
│                      ListOnlySession / DaemonPreamble        │
│                      （用 System.Threading.Channels 併發）    │
├──────────────────────────────┬──────────────────────────────┤
│  RsyncWin.Transport          │  RsyncWin.Fs                  │
│  IRsyncTransport 接縫：       │  Windows/NTFS 現實：          │
│  OpenSshProcessTransport      │  WindowsPathMapper、          │
│  DaemonTcpTransport           │  FileEnumerator、             │
│                              │  BasisFileStore、LocalTreePruner│
├──────────────────────────────┴──────────────────────────────┤
│  RsyncWin.Protocol  ★ 純核心，零 I/O ★                        │
│  Wire / Mux / Session / Checksums / FileList / Delta / 壓縮   │
└─────────────────────────────────────────────────────────────┘
```

**為什麼這樣分？** 因為 rsync 協定的正確性建立在一堆「錯一個位元組就靜默失敗」的公式上。把這些公式關進
一個零 I/O 的核心，就能用離線的黃金向量測試逐位元組驗證，不需要網路或 rsync 二進位檔。這是整個專案能成
立的承重決策。

角色引擎**永遠不直接判斷協定版本號**，而是讀取交握後產生的不可變 `SessionContext`（記錄協商到的協定
版本、compat flags、checksum 演算法、seed、壓縮方式、多工方向等）。

## 二、線路編解碼（Wire codecs）

這些是最底層、也最容易錯的位元組格式，全部在 `RsyncWin.Protocol/Wire/`：

### varint / varlong（`VarintCodec`）

rsync 的變長整數**不是 LEB128**。長度指示放在**第一個位元組**，由一張 `int_byte_extra` 表決定要再讀
幾個位元組；剩下的低位位元組是小端序（LSB first）。`varlong` 還帶一個 `min_bytes` 參數（不同欄位有不
同的最小長度，例如檔案大小是 3、mtime 是 4）。這張表是「表達」而非純量，依授權規範從寬鬆參考重新生成。

### write_ndx / read_ndx（`NdxCodec`）

檔案索引（ndx）有一套**自己的編碼**，不是單純的 int32：
- 相對於前一個值做差量壓縮。
- `0xFE` 是逃脫位元組。
- `0x80` 高位元有特殊意義。
- **正負索引各有獨立的執行狀態**。
- `NDX_DONE`（階段結束標記）在協定 ≥ 30 **不是** `write_int(-1)`，而是**單一位元組 `0x00`**。

這套編碼的狀態**跨整個 session 持續**（同一方向共用一個編碼器）。差異重傳（redo）時，重新請求 ndx 0 會
重新編成 `FE 00 00`（持久狀態逃脫形式），而不是全新的 `01`——這點被 `ssh31-pull-redo` 擷取釘死。

### vstring（`VstringCodec`）

協商字串的格式：1 個長度位元組（值 < 0x80），或高位元設定時的 2 位元組大端形式。checksum 與壓縮的演算法
清單就用這個格式交換。

### 多工（`Mux`：`MultiplexReader` / `MultiplexWriter`）

交握之後，串流被切成一個個 frame，每個 frame 有 4 位元組表頭（3 位元組長度 + 1 位元組 tag）。關鍵事實：
- **frame 邊界不帶語意**——一個邏輯值可以橫跨多個 frame。唯一正確的消費模型是「跨 frame 讀 N 個資料位
  元組」。
- 只有 `MSG_DATA`（tag 0）帶協定內容；其他 tag（Info/Error/Warning/Deleted/NoSend/IoError/ErrorExit…）
  是帶外訊息，交給處理器而非餵給協定讀取器。
- **零長度 `MSG_DATA` frame 是 keep-alive，不是 EOF**——把它當 EOF 是經典的卡住 bug。
- 多工方向是**不對稱且分版本**的：server→client 在每個我們支援的協定都多工；client→server 只在協定
  30+ 多工（29 寫原始位元組）。這些測得的方向記在 `SessionContext`。

## 三、Checksum：弱、強、與 seed 規則（`RsyncWin.Protocol/Checksums/`）

rsync 的差異演算法用兩種 checksum：

### 弱 checksum（`RollingChecksum`）

一個 Adler-32 變體的**滾動** checksum。滾動的意思是視窗往右移一格時，可以 O(1) 更新（減掉移出的位元組、
加上移入的位元組），不必重算整個視窗。這讓 sender 能在來源檔的**每一個位元組位置**都便宜地算出弱和。
（實作細節：rsync 用有號 char，這點若弄錯會全盤對不上。）

### 強 checksum（`StrongChecksum`）

可能是 MD4（舊、協定 29）、MD5，或協商到的 xxHash（xxh64/xxh3/xxh128）。三個容易錯的重點：

1. **每區塊強和被截斷到 `s2length`**；**整檔強和用完整長度**，而且可能用**不同的**演算法
   （`file_sum_nni` ≠ `xfer_sum_nni`）。
2. **seed 規則**：`seed == 0` 會短路所有 seed 混入。MD5 依 `CF_CHKSUM_SEED_FIX` 決定 seed 是前置還是
   後置；xxHash 對區塊用 `withSeed(seed)`，但對整檔用 `reset(0)`（整檔和忽略 session seed）。這個不對稱
   極易寫錯。
3. **xxHash 的位元組序**：`System.IO.Hashing` 輸出**大端**，rsync 是**小端**。必須經 `HashToUInt64` +
   `BinaryPrimitives` 做小端輸出，**絕不能直接 memcpy 雜湊 span**。

MD4 因為不在 BCL 裡，是依 RFC 1320 手工移植的（`Md4.cs`），並用 openssl legacy provider 的向量驗證。

## 四、區塊大小（`BlockSizer`）

`sum_sizes_sqroot` 決定差異傳輸的區塊長度 `blength`：
- 大約是 `sqrt(檔案大小)`，但**進位到 8 的倍數**（**不是** 2 的次方）。
- 下限 700（`BLOCK_SIZE`），上限 131072（`MAX_BLOCK_SIZE`）。
- `s2length = MIN(16, xfer_sum_len)`；redo 階段用完整長度。

這些用真 rsync 的 `--debug=deltasum2` 掃過 19 種檔案大小的輸出釘死。**不要憑記憶重建 `s2length`**——手推
的位元數學曾算出 1，而真 rsync 印的是 2。

---

## 五、檔案清單（`FileList`）

`FileListReader` / `FileListWriter` 是互為位元組反向的編解碼。每個 entry 依 `XMIT_*` 旗標決定哪些欄位
出現在線路上：

- **`SAME_NAME` + l1 前綴壓縮**：檔名與前一個共用的前綴長度用一個位元組表示，只傳後綴。
- **`SAME_TIME` / `SAME_MODE`**：值與前一個相同時省略。
- **`XMIT_MOD_NSEC`**：mtime 有非零次秒部分時，在 mtime varlong 與 mode 之間插一個 `varint(nsec)`。
  這讓 NTFS 的 100 ns tick 能安全往返（重傳不會因為次秒精度而誤判為變動）。
- **`F_SUM`（`--checksum`）**：僅一般檔，作為 entry 的**最後**一個欄位，16 位元組 xxh128，**不加 seed**
  （與整檔傳輸尾碼位元組相同）。

**排序是承重的**：`f_name_cmp` 是**序數位元組序**（含「頻帶規則」與虛擬結尾斜線）。任何文化相關或不分
大小寫的比較都會讓位置索引與伺服器失步。因此 `InvariantGlobalization` 全程開啟，檔名以原始位元組保存。

推送方向的 flist 由 `FileEnumerator` 走本機目錄產生 `FileEntry`，`FileListWriter` 依 reader 佈局反向
編碼；`--checksum` 時由 `PushSession.ComputeFlistChecksumsAsync` 串流計算每檔的 F_SUM 再附上（純核心不做
I/O，所以 checksum 由 Engine 層預先算好塞進 `FileEntry.FlistChecksum`）。

## 六、差異傳輸（Delta，`RsyncWin.Protocol/Delta/` — 最難的部分）

差異傳輸讓「幾乎相同的基準檔只傳有變動的區塊」。這是最難的階段——滾動 checksum 錯一個位元組就是靜默全量
重傳。

### 產生簽章（`SignatureGenerator`）

Generator（拉取時是我們、推送時是伺服器）把基準檔切成 `blength` 大小的區塊，對每塊算「弱和 + 截斷強和」，
連同一個 sum head（count / blength / s2length / remainder）送給 sender。

### 比對搜尋（`MatchSearcher` — sender 側）

Sender 拿到簽章後，對來源檔做**滾動搜尋**（`match.c:hash_search` 的行為）：
1. 維護一個 `blength` 大小視窗的滾動弱和。
2. 弱和命中時，**再驗證截斷強和**（弱和碰撞不算 match）。
3. 接受 match 時：先把待處理的字面位元組（literal）沖出（依 `CHUNK_SIZE = 32768` 切 token），再送出
   區塊參照 token，然後把掃描位置跳過整個 match 區域（跳躍會使滾動狀態失效，需重新計算）。
4. 沒 match 就往右滑一格（O(1) 滾動）。

`want_i` 相鄰偏好：當一個位置同時符合多個相同內容的區塊時，優先選「上一個 match 之後緊接的那塊」，讓一
連串重複區塊產生連續的參照（0,1,2,3）。

### Token 串流（`Token` / `FileReceiver`）

未壓縮的 token 是**純 4 位元組小端有號 int32**：
- `t > 0`：後面跟 t 個字面位元組（原始）。
- `t < 0`：參照基準區塊 `i = -(t+1)`，資料從**我們的**基準檔 `i * blength` 位移取。
- `t == 0`：本檔資料結束，緊接整檔 checksum 尾碼。

Receiver（`FileReceiver`）把 token 串流重建成檔案，邊寫邊算整檔 checksum。

### 整檔 checksum 尾碼與 redo

每檔結尾一定有整檔 checksum（用 `xfer_sum` 演算法、xxh128 為 seed 0、低 64 位小端接高 64 位小端）。
Receiver 算出的和與 sender 的尾碼**不符**時，代表基準檔在「簽章後、回覆前」的窗口內變了——這**不是**協定
錯誤，而是把該檔排進 **redo 清單**，在後續階段以完整長度簽章重傳一次。這正是為什麼協定有**多個階段與多個
`NDX_DONE` 標記**。第二次 redo 仍失敗才判為 exit 23。

**基準檔中途消失/縮短**時，`FileReceiver` 對缺的位元組**補零**（對齊 rsync `fileio.c` 的 `map_ptr` 行
為），讓整檔和自然對不上而走 redo，而不是讓整個 session 崩掉。

## 七、壓縮（`-z`，zlibx，`ZlibxTokenCodec`）

`-z` 時，純 int32 的 token 串流被換成一套 flag 位元組文法。RsyncWin **強制走 zlibx**（rsync 的「新」
壓縮）。詳見[文件四](04-與rsync銜接程度說明.md)的協定章節；這裡講演算法：

- **DEFLATED_DATA**（`(flag & 0xC0) == 0x40`）：長度 = `((flag & 0x3f) << 8) | 下一位元組`（14 位、
  無 +1），後跟該長度的原始 deflate 位元組。連續的 DEFLATED_DATA token（中間沒有 match token）組成**一個
  字面執行段（literal run）**，是**單一** deflate 串流，其結尾的 `00 00 ff ff`（Z_SYNC_FLUSH 標記）在
  線路上被**剝除**（解壓時要補回去）。
- **TOKEN_REL / TOKENRUN_REL**（`0x80` / `0xC0`）：區塊參照。`起始區塊 = 前一區塊游標 + delta`，游標從
  **0** 起算並推進到每個 run 的最後一塊；TOKENRUN 多帶 2 位元組執行數（涵蓋 `數值 + 1` 個連續區塊）。
- **END_FLAG**（`0x00`）：本檔結束，緊接**未壓縮**的 16 位元組整檔尾碼。

**能用 BCL 只靠 `DeflateStream` 實作的關鍵**：zlibx**不把符合區塊塞進 deflate 視窗**。因此 receiver 直接
從基準檔複製符合區塊，只把 DEFLATED_DATA 餵給 `DeflateStream`，永遠不需要一個「塞進視窗但不輸出」的原語
（`Z_INSERT_ONLY`，BCL 沒有）。編碼器（`MatchSearcher` 壓縮路徑）反之：把每個字面執行段用 `DeflateStream`
壓一次、`Flush()`（正好產生那個 `00 00 ff ff` 標記，再剝掉）、切成 DEFLATED_DATA token；符合區塊送
TOKEN_REL。若某個 match 的 delta 超出 6 位元組相對範圍，就退回當字面處理（避免用未擷取釘死的 TOKEN_LONG）。

整檔尾碼是硬安全網：任何重建歧異都會使雜湊不符 → redo → exit 23，**絕不會靜默損毀**。

## 八、編排引擎（`RsyncWin.Engine`）

### 併發模型

P4 早期版本「把某階段所有請求寫完才開始讀回覆」，在很大的檔案清單上會把雙向管線緩衝都填滿而互相卡死。P5
起改成**每階段兩個併發迴圈**：一個寫請求、一個讀回覆，透過一個有界的 `Channel<int>`（請求的 ndx）交接，
用 `Task.WhenAll` 併攏。任一迴圈失敗會取消連動 token，讓另一邊不會永遠等下去。

### 拉取（`PullSession`）— 我們是 receiver + generator

1. 交握 → 送空的過濾清單 → 收檔案清單。
2. **先把每個名字淨化映射**（`WindowsPathMapper`）**再碰檔案系統**：任何一個惡意名稱都要在什麼都沒建立
   的情況下中止。
3. Generator 對每個檔案決定：`-c` 時比對 F_SUM，否則走 mtime+大小快速路徑；需要傳輸就送出簽章請求。
4. Receiver 把回覆重建到**隨機命名的暫存檔**，整檔驗證後才 rename 到最終路徑；驗證失敗排 redo。
5. `--delete` 在傳輸尾巴後做本機修剪（`LocalTreePruner`）。

### 推送（`PushSession`）— 我們是 sender

1. 交握 → （`--delete` 時送空過濾清單）→ 送檔案清單（`--checksum` 時附 F_SUM）。
2. 讀 server-generator 的請求（ndx + iflags + sum head），跑 `MatchSearcher` 產生 token 串流 + 整檔尾碼。
3. 處理 server 的 redo 階段（依它的不符回報重送）。
4. `--delete` 時收 `MSG_DELETED` 帶外訊息並回報刪除。

### 常駐程式序幕（`DaemonPreamble`）

`rsync://` 連線先走**文字**序幕：`@RSYNCD: <ver>` 雙向問候 → 模組名 → `@RSYNCD: OK` 或
`AUTHREQD <challenge>`（MD5 挑戰-回應驗證）。OK 之後就是與 SSH 相同的多工 + 協定核心（只是跳過二進位版本
整數，因為版本已由文字問候協商）。

## 九、Windows / NTFS 現實（`RsyncWin.Fs`）

rsync 源自 Unix，檔名與屬性的假設常與 NTFS 對不上。`RsyncWin.Fs` 專門吸收這些差異：

- **`WindowsPathMapper`**：淨化/映射 Unix 檔名到合法 Windows 路徑——處理 `\`、`:`（磁碟機/ADS 冒號）、
  保留裝置名（CON、NUL、COM1…）、結尾點/空白、`\\?\` 長路徑。最後一道防線是解析後的路徑界內檢查（防目錄
  逃逸）。多對一映射時，flist 順序權威：第一個 entry 勝出，衝突者整個排除（含其子項）。
- **`FileEnumerator`**：走本機目錄產生 `FileEntry`，並**預先排成 rsync 的 flist 序**。
- **`BasisFileStore`**：差異傳輸時對基準檔做隨機讀取。
- **`LocalTreePruner`**：`--delete` 拉取時的本機修剪（reparse point 安全、界內檢查、清唯讀屬性）。
- **mtime 夾制**：線路上的 mtime 夾到可設定的 Win32 FileTime 範圍
  （1601-01-01…9999-12-31），而不是丟例外中止整個 session。
- **暫存檔**：以隨機 8 hex 命名（`.<final>.<8-hex>.rsyncwin-tmp`），避免與既有使用者檔案衝突。

## 十、測試策略

正確性由**第一層閘門**把關，它不需要網路：

1. **隱密黃金向量**（`RsyncWin.Protocol.Tests` 等）：從真 rsync 擷取的位元組存進 `test-fixtures/`，重播
   給純核心。checksum seed 是時間相依的，所以測試**從擷取讀出 seed**，絕不假設。
2. **Live 互通**（`[Trait("Category","Interop")]`）：對真 rsync（Docker 容器）跑端對端，每個測試都有卡住
   偵測逾時（階段邊界 bug 表現為卡住而非失敗）。

閘門永遠是位元組/雜湊相等或「重跑不傳輸任何東西」。目前 **418 個隱密測試 + 29 個 live 互通測試全綠**
（詳見 [`docs/acceptance-report.md`](../acceptance-report.md)）。
