# RsyncWin 使用說明

本文說明 `rsyncwin` 命令列工具的所有用法。RsyncWin 是純命令列工具，行為與參數盡量對齊標準 rsync，讓
熟悉 rsync 的使用者能無痛上手。

## 建置與執行

```powershell
# 建置
dotnet build

# 直接執行
dotnet run --project src/RsyncWin.Cli -- <參數...>

# 或建置後直接跑產出的 rsyncwin.exe
```

## 基本語法

`rsyncwin` 一定是「一個本機路徑 + 一個遠端路徑」，方向由**哪一邊是遠端**決定：

- 遠端在**來源**（第一個參數）→ **拉取（pull）**：從遠端同步到本機。
- 遠端在**目的**（第二個參數）→ **推送（push）**：從本機同步到遠端。

遠端路徑有三種寫法：

| 形式 | 意義 |
|---|---|
| `[user@]host:path` | 透過 SSH（單一冒號） |
| `rsync://[user@]host[:port]/module[/path]` | 透過 rsync 常駐程式（可指定埠，預設 873） |
| `[user@]host::module[/path]` | 同上的簡寫（埠固定 873） |

> **Windows 路徑不會被誤判**：`D:\backup`、`D:/backup`、`\\server\share`（磁碟機代號或 UNC）一律視為
> 本機路徑，不會被單冒號規則當成遠端主機。

## 常用範例

### SSH 拉取

```powershell
# 從遠端拉整個目錄樹（遞迴 + 保留時間）到本機
rsyncwin -rt user@example.com:/srv/data/ D:\backup\data

# 封存模式（-a：遞迴、保留時間/連結/擁有者/群組/裝置/權限）
rsyncwin -a user@example.com:/srv/data/ D:\backup\data
```

### SSH 推送

```powershell
# 把本機目錄推到遠端（推送目前支援 -rt，不支援 -a 的額外欄位）
rsyncwin -rt D:\projects\site\ user@example.com:/var/www/site/
```

### rsync 常駐程式（daemon）

```powershell
# 匿名拉取
rsyncwin -rt rsync://mirror.example.com/debian/ D:\mirror\debian

# 需要驗證：密碼放在 RSYNC_PASSWORD 環境變數
$env:RSYNC_PASSWORD = "opensesame"
rsyncwin -rt rsync://alice@host/secret/ D:\secret

# 推送到可寫模組
rsyncwin -rt D:\upload\ rsync://host/incoming/

# 列出常駐程式提供的模組（模組名留空）
rsyncwin rsync://mirror.example.com/
```

## 旗標一覽

| 旗標 | 長格式 | 說明 | pull | push |
|---|---|---|:--:|:--:|
| `-r` | `--recursive` | 遞迴子目錄 | ✅ | ✅ |
| `-t` | `--times` | 保留修改時間（預設即開啟） | ✅ | ✅ |
| `-a` | `--archive` | 封存模式（含 -r -t 及連結/擁有者/群組/裝置/權限） | ✅ | ❌ |
| `-c` | `--checksum` | 以整檔 checksum 比對，而非 mtime+大小 | ✅ | ✅ |
| | `--delete` | 刪除目的端多出來、來源端沒有的檔案 | ✅ | ✅ |
| `-s` | `--secluded-args`（別名 `--protect-args`） | 保護含空白/特殊字元的遠端路徑 | ✅ | ✅ |
| `-z` | `--compress` | 用 zlibx 壓縮傳輸的 token 串流 | ✅ | ✅ |
| `-e` | `--rsh` | 指定要用的遠端 shell（預設內建 ssh.exe） | ✅ | ✅ |

短旗標可以合併：`-rtcz`、`-rtz --delete` 等。

### 各旗標的重點行為

- **`-c` / `--checksum`**：跳過「mtime + 大小」快速路徑，改用整檔 checksum 比對。因此即使大小與時間都
  相同、但內容不同的檔案，也會被正確地重新傳輸；反之內容相同、只有時間不同的檔案只會修時間、不傳資料。
- **`--delete`**：需要搭配 `-r` 或 `-a`（rsync 不允許不遞迴就刪除）。
  - **拉取時**是純本機動作：掃描目的端，刪掉來源端沒有的項目（reparse point 安全、含界內檢查、會清除
    唯讀屬性），且若檔案清單有 io_error 就跳過刪除。
  - **推送時**由遠端接收端執行刪除，並用 `MSG_DELETED` 逐項回報，本機以 `deleting <path>` 印出。
- **`-s` / `--secluded-args`**：一般情況下，遠端路徑會被遠端的 shell 拆解，含空白的路徑（如
  `/data/my folder/`）會被拆成兩個參數而失敗。加上 `-s` 後，檔案參數改在版本交握之前以 NUL 分隔的清單
  傳送，遠端 rsync 直接讀取，空白原樣保留。SSH 傳輸才需要；常駐程式本來就以 NUL 分隔參數，天生安全。
- **`-z` / `--compress`**：使用 **zlibx** 壓縮。標準 rsync 的 `-z` 預設會協商到 `zstd`，但 .NET 的 BCL
  只有 deflate，所以 RsyncWin 強制走 zlibx（透過 `--new-compress`）。對高度可壓縮的內容（文字、重複資
  料）能顯著減少傳輸量；對不可壓縮內容（已壓縮檔）則自動走 stored block，不會膨脹。

## 離開碼（exit code）

RsyncWin **原封不動回傳 rsync 的數值離開碼**，方便腳本判斷：

| 碼 | 意義 |
|---|---|
| 0 | 成功 |
| 1 | 語法/用法錯誤 |
| 2 | 協定不相容 |
| 4 | 請求了不支援的動作（如無共同 checksum 演算法） |
| 5 | 啟動 client-server 失敗（含 ssh.exe 以 255 離開時：主機無法連線、驗證失敗等） |
| 10 | Socket I/O 錯誤（常駐程式連線被拒/DNS 失敗） |
| 11 | 檔案 I/O 錯誤（Windows 檔案系統失敗歸此類，而非 12） |
| 12 | 協定串流錯誤/失步 |
| 23 | 部分傳輸失敗（有檔案沒傳成功、或遠端回報錯誤） |
| 24 | 來源檔在傳輸中途消失 |
| 30 | I/O 逾時 |

> **重要細節**：ssh.exe 本身以 255 離開（SSH 層失敗）會回報 **5**，而非 12——這是刻意的排序修正，讓
> 「主機無法連線」正確對應到 rsync 的語意。

## 密碼與憑證

- **常駐程式驗證**：密碼從 `RSYNC_PASSWORD` 環境變數讀取。RsyncWin **不會**互動式詢問密碼，也不會把密碼
  寫進命令列（避免出現在行程清單）。
- **SSH 驗證**：完全交給 `ssh.exe`——請用你平常的 SSH 金鑰/agent 設定。RsyncWin 以 `BatchMode` 傾向
  非互動；若需要輸入密碼，請先設定好金鑰或 agent。

## 疑難排解

- **卡住不動**：rsync 協定的階段邊界 bug 表現為卡住而非報錯。若發生，通常是對端版本或旗標組合超出目前
  支援範圍。所有內部路徑都有逾時保護。
- **`exit 5` 且印出 ssh 錯誤**：SSH 連線本身失敗（金鑰、主機、網路）。錯誤訊息是 ssh.exe 的 stderr。
- **`exit 2` 「protocol >= 30 required」**：對端是 rsync 3.1.x 或更舊（協定 29/30 的檔案清單格式未實
  作）。RsyncWin 針對協定 31 實作，向下協商到 30。
- **含空白的遠端路徑失敗**：加上 `-s`。
